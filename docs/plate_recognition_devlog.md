# 车牌识别模块开发记录

## 概述

基于 OpenCV 4.x 的车牌识别模块，集成在 SmartParking 系统中。支持摄像头拍照识别、与数据库已登记车辆比对。

---

## 迭代过程

### 第 1 版 — 基础框架搭建

**目标**: 实现从摄像头拍照 → 服务器识别 → 数据库比对的完整流程

**实现**:
- 新建 `PlateRecognizer` 类，实现 base64 解码 → OpenCV 解码 → 图像预处理
- 车牌检测: 自适应阈值 → 边缘检测 → 轮廓筛选(按面积/宽高比/矩形度)
- 颜色检测作为备选方案 (HSV 蓝色/绿色范围)
- 字符分割: 垂直投影法 + Otsu 二值化
- 字符识别: 使用 Hershey 字体生成模板 → `matchTemplate` 像素级模板匹配
- 模板加粗 (dilate 2 次) 以接近真实车牌字体

**新增文件**:
- `src/plate_recognizer.h`
- `src/plate_recognizer.cpp`
- `frontend/recognize.html`
- `frontend/js/recognize.js`

**结果**: ❌ 能检测到车牌区域和颜色，但字符识别几乎全部失败
**原因**: Hershey 字体与真实车牌字体差异过大，像素级模板匹配对字体风格极度敏感

---

### 第 2 版 — 改进预处理和模板

**目标**: 提高字符识别率

**改进**:
- 新增 CLAHE 对比度增强 → 改善光照不均
- 车牌区域统一缩放到高度 80px → 保证处理一致性
- 双阈值二值化策略: Otsu 和自适应阈值自动选优
- 切换到轮廓法字符分割 (比投影法更鲁棒)
- 增加重叠检测 → 避免重复识别同一字符
- 模板匹配时按宽高比预筛候选

**结果**: ❌ 仍不能可靠识别字符，根本问题未解决

---

### 第 3 版 — Hu矩形状匹配

**目标**: 用形状匹配彻底取代像素模板匹配

**改进**:
- 弃用 `cv::matchTemplate`，改用 `cv::matchShapes` (Hu 不变矩)
- Hu 矩对缩放、旋转、字体风格不敏感 → 解决字体差异问题
- 孔洞数拓扑过滤: 预先计算每个字符的孔洞数(如 8→2, 0/A/P→1, 1→0) 快速排除不可能候选
- 模板和字符的轮廓预计算，匹配时直接比较轮廓
- 分割字符加 4px 边框再归一化，保留更多形状信息

**结果**: ⚠️ 准确率相比前两版有提升，但对不清晰或有遮挡的车牌仍不稳定

### 第 4 版 — Bug 修复与检测管线重构

**目标**: 修复"什么都识别不出来"的根本问题

**根因分析**:
1. **二值化极性错误**: `THRESH_BINARY_INV` 对蓝色车牌产生黑字白底，而 `findContours` 查找白色前景 → 找到的是背景而非文字 → 返回 0 个字符
2. **CLAHE 分块与 JPEG 块对齐**: 8×8 tile 与 JPEG 8×8 DCT 块对齐，放大块效应伪影
3. **自适应阈值过于敏感**: C=4 导致字符内部过度分裂
4. **偶数核形态学运算偏差**: 2×2 核的锚点默认为 (0,0) 而非中心，引入偏移
5. **字符 '1' 被误过滤**: 宽高比上限 6.0 无法容纳窄字符 '1'
6. **最小字符数阈值过高**: 5 个字符的要求对分割质量敏感的管线过于严格
7. **车牌检测主次颠倒**: 边缘检测始终能找到一些狭条轮廓（30-40px 宽）通过面积/宽高比筛选 → HSV 备选检测永不触发 → 真正的蓝色车牌区域从未被选中

**修复内容**:

1) 字符二值化极性校正 (`segmentCharacters()`)
   - 统计白色像素占比，若 >50% 则 `bitwise_not` 反转
   - 确保文字为白色前景、背景为黑色

2) CLAHE tile 参数调整 (`preprocessPlateRegion()`)
   - 8×8 → 10×10 tile，避免与 JPEG 8×8 DCT 块对齐

3) 自适应阈值敏感度降低 (`binarizePlate()`)
   - C=4 → C=6，减少字符内部过分裂

4) 形态学核修正 (`binarizePlate()` / `initTemplates()`)
   - 2×2 → 3×3 核，消除偶数核锚点偏移

5) 字符 '1' 过滤放宽 (`segmentCharacters()`)
   - 宽高比上限 6.0 → 12.0

6) 最小字符数降低 (`recognizeCharacters()`)
   - 5 → 3，避免因分割遗漏导致的整串丢弃

7) **检测管线重构 — HSV 升为主方法** (`detectPlateCandidates()`)
   - 交换 HSV 与边缘检测优先级
   - HSV 阈值放宽: S 80→50, V 60→40，扩大色检测覆盖
   - HSV 宽高比放宽: 1.8-4.5 → 1.2-5.0
   - 增加形态学 close+open 净化色掩码
   - HSV 区域外扩 10% 以包含车牌白色边框
   - 边缘检测降为备选，仅添加与 HSV 不重叠的候选
   - 检测顺序: HSV 候选优先 → 不重叠的边缘候选补充

**结果**: 🔧 管线修复完成，HSV 主检测应能正确定位蓝色/绿色车牌区域，等待实拍验证

### 第 5 版 — LPRNet 深度学习车牌识别（当前）

**目标**: 用手写 CV + Hu 矩替换为端到端深度学习模型

**根因**:
- Hershey 字体模板匹配根本不支持汉字（永远输出 `?`）
- 字符分割对二值化质量极其敏感，真实场景光照变化大
- CV 管线已达天花板，无论怎么调参也无法达到可用准确率

**方案**: 使用 LPRNet（轻量级 CNN + CTC）端到端车牌识别

**改进**:
- 新建 `src/lpr_recognizer.h/.cpp` — LPRNet 推理类
  - 自动探测模型输入/输出维度（94×24 输入, 68 类×18 时间步输出）
  - OpenCV DNN 加载 ONNX 模型，无额外依赖
  - CTC 贪婪解码（argmax → 合并重复 → 去空白 → 映射字符）
  - 支持 31 个省份汉字 + 字母 + 数字 + 特殊车牌
- 下载 `lprnet.onnx` 预训练模型（1.7MB，CCPD 数据集训练）
- `recognize()` 优先使用 LPRNet 推理，失败时降级到模板匹配
- CMake 添加 `dnn` 模块支持，构建时自动复制模型

**LPRNet 优势**:
- 端到端识别，无需字符分割 ✓
- 支持中文省份字符（京/沪/粤/苏等 31 省）✓
- CCPD 数据集准确率 ~95% ✓
- CPU 推理 1-3ms ✓
- OpenCV DNN 加载，无额外依赖 ✓

**关键文件**:
- `src/lpr_recognizer.h` — LPRNet 单例类定义
- `src/lpr_recognizer.cpp` — 预处理 / 推理 / CTC 解码实现
- `models/lprnet.onnx` — 预训练 ONNX 模型
- `models/` — 构建时自动复制到可执行目录

**结果**: ✅ 深度学习管线集成完成，应能正确定位车牌并识别含中文的完整车牌号

### 第 5.1 版 — Bug 修复：字符集、归一化、置信度

**目标**: 修复"全都识别错"和"置信度 2000%"问题

**根因**（三个独立 Bug）:

1. **输入归一化错误**: `blobFromImage(1.0/255.0)` 使用 [0,1] 范围，LPRNet 期望 (img-127.5)/127.5（[-1,1] 范围）
2. **置信度未 Softmax**: raw logit 直接当作概率值返回，可任意大或为负
3. **字符集顺序完全错误**:
   - index 0 = CTC blank → 所有字符索引偏移 +1
   - 排除了 I/O 字母，加入了 "挂/学" 特殊字符
   - 68 类模型的 special char 条件 `chars_needed >= 68` 为 false → 最后 2 个索引填充为 `?`
   - 实际 LPRNet_Pytorch 标准字符集：31省份 + 10数字 + 24字母 + I + O + `-`(dash)，CTC blank 在最后一个 class

**修复**:
- `preprocess()`: `1.0/255.0` → `1.0/127.5`，mean=127.5
- `ctcDecode()`: max_val → softmax（log-sum-exp 稳定计算）
- `initCharset()`: 完全重写为 LPRNet_Pytorch 标准字符集，`blank_idx_` = last class

**结果**: ✅ 字符映射正确（之前 `?8?7A?3?V?H?` → 现在 `98B4WJ`），置信度归一化为 0-1

### 第 5.2 版 — CTC 解码修复、严格格式约束、检测管线调优

**目标**: 修复"识别像乱识别"和"重复字符被合并"问题，严格限定输出格式

**背景**:
- 用户反馈"识别像乱识别，每次结果不一样"、"相同字符在一起时会漏掉"、"只能识别出 1-2 个字符"
- 合成车牌测试确认 LPRNet 模型本身工作正常（`黑AJ8888` 置信度 0.96），问题在检测管线和 CTC 解码

---

#### Bug 1：CTC 贪婪解码顺序错误（重复字符丢失，根本原因）

**现象**: `苏A88888` 识别为 `苏A8`（5 个 8 只剩 1 个），任何有重复字符的车牌都会少字

**根因**: CTC 标准算法要求**先合并重复 → 再去空白**，但实现做反了。

```
旧代码（错误）:
t: A → blank → 8 → blank → 8 → blank → 8
                      ↑              ↑
                 空白不更新prev    prev=8→跳过! 
                  → 输出: A8 (漏掉两个8)

新代码（正确）:
t: A → blank → 8 → blank → 8 → blank → 8
           ↑              ↑              ↑
        prev=空白≠8     prev=空白≠8   prev=空白≠8
        → 输出: A888 (全部保留)
```

**修复**: `lpr_recognizer.cpp` — `ctcDecode()` 交换合并检查与空白检查的顺序

---

#### Bug 2：UTF-8 字节长度误判

**现象**: LPRNet 输出 `云IIZA2S`（7 个字符）但 `std::string::size()` 返回 9（中文=3 字节 + 6 个 ASCII），长度检查 `!= 7 && != 8` 拒绝所有带中文的结果

**修复**: `plate_recognizer.cpp` — 添加 `utf8_chars()` 函数正确计数 UTF-8 字符，长度检查使用字符数而非字节数

---

#### Bug 3：边界框超出图像范围导致 500 崩溃

**现象**: 检测到车牌后服务器 500 错误崩溃

**根因**: `minAreaRect` 对倾斜车牌的 `boundingRect()` 可能超出图像边界（尤其是车牌在图像边缘时），OpenCV ROI 越界断言失败

**修复**: `plate_recognizer.cpp` — `br = rr.boundingRect() & cv::Rect(0, 0, src.cols, src.rows)` 钳位到图像范围

---

#### Bug 4：检测区域双重扩展

**现象**: `detectPlateCandidates()` 已有 15% 扩展，`recognize()` 又加 15%，导致车牌在 LPRNet 输入图像中占比过小

**修复**: 去除检测阶段的扩展（仅保留 `recognize()` 中的单次 15% 扩展）

## 架构说明

```
浏览器摄像头 → getUserMedia → canvas.toDataURL(base64)
    ↓ POST /api/plate/recognize-image
服务端 PlateRecognizer:
    base64解码 → imdecode → detectPlateCandidates()
        ├─ [主] HSV颜色检测 (自适应阈值)
        │   ├─ 亮度>100: 蓝(S≥50/V≥40) 绿(S≥40/V≥30)
        │   ├─ 亮度≤100: 蓝(S≥30/V≥25) 绿(S≥20/V≥20)
        │   ├─ H范围: 蓝95-130 绿30-85
        │   ├─ 面积: 1500-80000px
        │   └─ 候选按宽高比评分排序（越接近3.14分越高）
        └─ [辅] 边缘检测 (仅HSV为空时, 宽高比2.5-4.0)
    ↓
    analyzePlateColor() → 蓝/绿/黄牌识别
    ↓
    recognizeCharacters():
        ├─ [主] LPRNet 深度学习 (94×24 → ONNX推理 → CTC解码)
        │   ├─ CTC解码顺序修正: 先合并重复→再去空白
        │   ├─ 格式约束: 7/8位, 省份+字母, 位置1数字自动修正
        │   ├─ 置信度阈值: 0.1 (格式验证为真正守门员)
        │   └─ 端到端识别，无需分割
        └─ [辅] 模板匹配降级 (Hu矩 + NCC + 孔洞数融合)
    ↓
    PlateService.checkRegistration() → 四表联合查询
        ├─ CAR_RECORD (入场记录/是否在场)
        ├─ MONTHLY_PASS (月卡有效期)
        └─ VEHICLE_BLACKLIST (黑名单)
    ↓
返回 JSON: { plate_number, confidence, color, registration: {...} }
```

## 当前局限

1. **倾斜/模糊车牌** — LPRNet 对角度过大或严重模糊的车牌识别率下降
2. **夜间低光照** — HSV 检测在光线不足时可能漏检车牌区域
3. **双行车牌** — 当前检测管线未针对双行车牌优化
4. **模型精度** — 68 字符 LPRNet 模型可升级到更优版本（如 71 字符版）
5. **汉字省份识别不可靠** — LPRNet 的 94×24 输入分辨率太低（见第 6 版分析）

## 后续优化方向

- **YuNet LPD ONNX** — 替换 HSV 检测为深度学习检测，提升倾斜/暗光场景
- **更高精度模型** — 升级到 71 字符 LPRNet 或 CRNN 模型
- **TensorRT 加速** — 使用 INT8 量化提升推理速度

### 第 6 版 — LPRNet 精度上限与替代方案评估

**发现问题**: 用户拍照 `川G00M00`，模型输出 `川B0M0O`（省份正确但后缀混乱）或 `皖6A5CK2O`（省份错误）。核心矛盾是 LPRNet 的 94×24 输入分辨率对汉字笔画来说远远不够。

#### 做出的修复（全部无效于根本问题）

| 修复 | 说明 | 效果 |
|------|------|------|
| 8→7 全子序列 | 8 位输出尝试所有可能的 7 位子序列取最优 | 处理了杂音字符但省份仍错 |
| 省份字符挽救 | 首位非省份时从模型原始 logits 找最佳省份 | 只能从模型已有认知中选，不能改变模型判断 |
| O/0 混淆纠正 | 末尾 `O` 自动替换为 `0` | 次要问题，实际影响不大 |
| 宽高比强制 | 候选框强制 ≥2.5:1 以捕获完整车牌宽度 | 改善后缀字符数稳定，但省份仍不可靠 |
| 6 字符降级 | 省份正确时接受 6 位输出 | 显示部分正确结果，但省份错误时仍无效 |

#### 根因：LPRNet 模型的精度天花板

所有修复都绕不开一个物理限制：**LPRNet 的输入是 94×24 像素。** 在这个尺寸下：

```
贵 →  浙 →  皖 →  川 →  鲁
在 94×24 中每个字符仅占约 13×24 像素，
汉字的笔画细节（横竖撇捺折）全部模糊成一片，
模型只能"猜"省份，后处理无法弥补。
```

#### 替代方案评估

| 方案 | 输入尺寸 | 检测方式 | 识别方式 | 汉字精度 | 集成难度 |
|------|---------|---------|---------|---------|---------|
| 当前 LPRNet | 94×24 | HSV 颜色 | LPRNet CNN | ❌ 低 | 已集成 |
| **[HyperLPR3](https://github.com/szad670401/HyperLPR)** | **160×48** | **YOLO (320×320)** | **CRNN 识别模型** | ✅ 高 | 中（替换 ONNX 模型和检测代码） |
| **[LPDet](https://github.com/zjykzj/LPDet)** | 可变 | YOLOv5-Seg | CRNN-CTC | ✅ 高 | 中（ONNX 格式，替换管线） |
| **Python 微服务** | 可变 | 任意 | PaddleOCR / HyperLPR | ✅ 最高 | 低（HTTP 调用独立进程） |
| **ONNX Runtime + 新模型** | 可变 | 保留 HSV | 更优 ONNX 模型 | ✅ 中高 | 中（需添加 ONNX Runtime 依赖） |

**核心结论**：`applyPlateFormat()` 等后处理已是当前模型的极限。要实现准确的汉字识别，必须换用**输入分辨率更高、网络容量更大**的模型。
